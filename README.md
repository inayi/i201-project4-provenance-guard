# Provenance Guard

A backend any creative-sharing platform can plug into to judge whether submitted text
was written by a human or generated by AI, score confidence honestly, surface a
plain-language transparency label, and let creators appeal.

The system is built around one principle: **on a writing platform, wrongly labeling a
human's work as AI is the worst possible error.** So it demands corroborated evidence
before ever calling something AI, communicates uncertainty instead of forcing a binary,
and gives every creator a path to contest a decision.

> Design rationale and the full architecture diagram live in [planning.md](planning.md).

## Quick start

```bash
python -m venv .venv && .venv/Scripts/activate     # Windows; or: source .venv/bin/activate
pip install -r requirements.txt
echo "GROQ_API_KEY=your_key_here" > .env           # enables the LLM signal
python app.py                                       # http://127.0.0.1:5000
```

| Method & path | Purpose |
|---------------|---------|
| `POST /submit` | Classify content (`creator_id` + `text`, or `content_type:"image_metadata"` + `metadata`); returns attribution, confidence, label. |
| `POST /appeal` | Contest a classification (`content_id` + `creator_reasoning`); sets status `under_review`. |
| `POST /verify` | Earn a "Verified Human" certificate (`creator_id` + `attestation`). |
| `GET /analytics` | Detection-pattern + platform-health dashboard. |
| `GET /content/<id>` | Fetch a stored decision. |
| `GET /log` | Structured audit log (`?limit=N`). |
| `GET /health` | Liveness + whether the LLM signal is configured. |

## Architecture overview — the path a submission takes

1. **`POST /submit`** receives `{ "text", "creator_id" }`.
2. **Rate limiter** (flask-limiter, keyed per `creator_id`) admits the request if the
   creator is under 10/min and 100/day.
3. **Detection pipeline** (`detection.evaluate_content`) hands the *same* text to three
   independent signals, each returning an AI-likeness probability in `[0,1]`:
   burstiness, repetition, and an LLM judge.
4. **Ensemble fusion** combines them: `p_ai = 0.50·llm + 0.25·burstiness + 0.25·repetition`.
5. **Label mapping** turns `p_ai` into an attribution + the exact transparency-label text
   using asymmetric thresholds (`≥0.80` AI, `0.40–0.80` uncertain, `<0.40` human).
6. **Audit log** (SQLite) writes one structured row: content_id, creator_id, timestamp,
   all three raw signal scores, `final_p_ai`, attribution, confidence, and status.
7. **Response** returns `content_id`, `attribution`, `confidence`, `label`, and the
   per-signal breakdown. A creator can later `POST /appeal` with the `content_id`.

## Detection signals — what each measures, why, and what it misses

Three **independent** signals; independence is the point — each is foolable alone, so we
require corroboration before accusing anyone.

### 1. Burstiness (sentence-length coefficient of variation)
- **Measures:** `CV = std_dev / mean` of per-sentence word counts, mapped to
  `ai_p = clamp(1 − CV/0.9)`.
- **Why:** humans write with jagged rhythm (short + long sentences → high CV); default
  LLM prose is metronomically uniform → low CV → higher AI probability.
- **Misses:** uniform-by-design human forms — haiku, list poems, structured stanzas —
  look "AI-uniform." Texts under 2 sentences give it nothing, so it abstains at 0.5.

### 2. Repetition (composite: TTR + repeated openers + duplicate bigrams)
- **Measures:** type-token ratio (vocabulary diversity), the most-common two-word
  sentence opener as a fraction of sentences, and the duplicate-bigram fraction,
  combined `0.45·diversity_penalty + 0.35·opener_repeat + 0.20·bigram_repeat`.
- **Why:** LLMs lean on formulaic scaffolding ("First… Second… Third…") and recycled
  phrasing that burstiness misses when sentence *lengths* happen to vary.
- **Misses:** deliberately repetitive human art — refrains, choruses, anaphora ("I have
  a dream…") — reads as formulaic despite being human.

### 3. LLM judge (Groq `llama-3.3-70b-versatile`)
- **Measures:** a deterministic (`temperature=0`) call asks the model for
  `{"ai_probability", "reason"}` as JSON, weighing idiosyncrasy, texture, and uniformity.
- **Why:** the strongest single signal — it catches semantic/stylistic tells the cheap
  statistics can't, so it carries half the ensemble weight.
- **Misses:** hyper-polished human writing (academic abstracts, heavily edited corporate
  prose) can read as AI-like. **Restrictive fallback:** on missing key / API error /
  unparseable JSON it logs loudly and returns **0.0** (not 0.5) — a broken LLM can only
  pull `p_ai` toward "human," never manufacture a false accusation.

## Confidence scoring — how signals combine, and that it's meaningful

The three probabilities fuse with fixed, documented weights:

```
p_ai = (0.50 × llm_score) + (0.25 × burstiness_score) + (0.25 × repetition_score)
```

`confidence = max(p_ai, 1 − p_ai)` — certainty in whichever direction the score leans:
high for a clear human or clear AI verdict, ~0.5 for genuinely borderline text.

Attribution is assigned with **asymmetric thresholds** (verified against
[planning.md](planning.md)):

| `p_ai` | attribution |
|--------|-------------|
| `≥ 0.80` | `likely_ai` |
| `0.40 – 0.80` | `uncertain` |
| `< 0.40` | `likely_human` |

The `0.80` AI bar is deliberately high: since the LLM contributes at most `0.50`, a
`likely_ai` verdict **requires the statistical signals to agree** — corroboration is
mandatory before the system points a finger. This directly encodes the asymmetric risk
that falsely accusing a human is catastrophic.

### Validation — 4 deliberately chosen inputs (Milestone 4 testing)

| Input | attribution | p_ai | burst / rep / llm |
|-------|-------------|------|-------------------|
| Clearly AI (formal essay) | `uncertain` | 0.546 | 0.58 / 0.00 / 0.80 |
| Clearly human (casual ramen review) | `likely_human` | 0.180 | 0.32 / 0.00 / 0.20 |
| Borderline: formal human (economics) | `uncertain` | 0.579 | 0.72 / 0.00 / 0.80 |
| Borderline: lightly edited AI (remote work) | `uncertain` | 0.494 | 0.58 / 0.00 / 0.70 |

The casual human piece scores far lower than everything else, and the borderline cases
land squarely in the middle — the score varies meaningfully and reaches all three
categories. (Note the "clearly AI" essay lands `uncertain`, not `likely_ai`: its
sentence lengths vary enough that burstiness disagrees with the LLM, so the ensemble
withholds an accusation. That is the false-positive-averse design working, and it is
called out in Known Limitations.)

### Two example submissions with noticeably different confidence

**High-confidence case** — repetitive, template text:

```json
{ "attribution": "likely_ai", "confidence": 0.9606, "final_p_ai": 0.9606,
  "signals": { "burstiness_score": 1.0, "repetition_score": 0.8624, "llm_score": 0.99 } }
```

**Lower-confidence case** — lightly edited AI about remote work:

```json
{ "attribution": "uncertain", "confidence": 0.5061, "final_p_ai": 0.4939,
  "signals": { "burstiness_score": 0.5755, "repetition_score": 0.0, "llm_score": 0.7 } }
```

`0.96` vs `0.51` confidence, across two different label categories — the scoring
produces real variation, not a constant.

## Transparency label — all three variants (verbatim)

The label returned by `/submit` changes with the score. Exact text:

| Variant | Label text |
|---------|------------|
| **High-confidence AI** | 🤖 Likely AI-generated. Our analysis suggests this content was probably produced with the help of an AI tool. This is an automated estimate, not a certainty. The creator can appeal this label. |
| **High-confidence human** | ✍️ Likely human-written. Our analysis found no strong signs of AI generation. This is an automated estimate, not a guarantee of authorship. |
| **Uncertain** | ❔ Authorship uncertain. Our analysis could not confidently determine whether this content was written by a human or generated by AI. We are showing this honestly rather than guessing. Treat the work on its own merits. |

Every label states it is an *estimate*, not a verdict; the AI label always advertises
the right to appeal; the uncertain label frames "we don't know" as honest, not punitive.

## Appeals workflow

`POST /appeal` with `{ "content_id", "creator_reasoning" }`:
1. Looks up the original decision by `content_id` (404 if unknown).
2. Sets its `status` to `under_review` and stores the reasoning in `appeal_reasoning` —
   the original scores and attribution are **never overwritten**.
3. Returns confirmation echoing the original decision. (No automated re-classification —
   the appeal routes a contested case to a human reviewer.)

```bash
curl -s -X POST http://localhost:5000/appeal -H "Content-Type: application/json" \
  -d '{"content_id": "b95730af-...", "creator_reasoning": "I wrote this myself from personal experience. I am a non-native English speaker and my writing style may appear more formal than typical."}'
```
```json
{ "content_id": "b95730af-...", "status": "under_review",
  "appeal_reasoning": "I wrote this myself from personal experience...",
  "original_attribution": "likely_ai", "original_confidence": 0.9606,
  "message": "Your appeal has been logged. This content is now under review." }
```

## Rate limiting

flask-limiter, keyed per creator (`creator_id` from the body, then `X-Creator-Id`
header, then IP). Applied to `POST /submit`:

| Scope | Limit | Reasoning |
|-------|-------|-----------|
| `POST /submit` | **10 / minute** | A real creator submits a poem or post occasionally, then waits — 10/min covers revising and resubmitting a few pieces while stopping a script from hammering the paid, LLM-backed endpoint. |
| `POST /submit` | **100 / day** | Even a prolific writer rarely produces 100 distinct works a day. This is the anti-abuse ceiling: it caps LLM cost and blunts a scraper probing thousands of inputs to reverse-engineer the detector. |
| `POST /appeal` | **5 / minute** | Appeals are deliberate, reasoned actions — 5/min stops appeal-spam without obstructing a genuine creator. |

The numbers sit well above realistic human usage and well below what a flood or a
decision-boundary-mapping attack needs.

## Known limitations

**Structured, repetitive human poetry (haiku, villanelles, list poems, refrains) will
be misclassified toward AI.** This is not a data problem — it is a direct consequence of
two of our three signals. Uniform line lengths spike the *burstiness* score (low CV
reads as machine-uniform) and a repeated refrain spikes the *repetition* score. On a
genuinely human poem, two of three statistical signals fire in the wrong direction; only
the LLM judge is likely to recognize it as human creative writing. The `0.80` threshold
and the LLM's 50% weight are what keep most such cases in `uncertain` rather than
`likely_ai`, but a short, highly regular poem is exactly the content this design is
weakest on.

**Conversely, the "clearly AI" formal essay in our test set scored `uncertain`, not
`likely_ai`** — its sentence lengths varied enough that burstiness disagreed with the
LLM. The system errs toward *not* accusing, which means polished AI prose with human-like
rhythm can slip into `uncertain`. That is the deliberate trade-off: we accept more false
negatives to avoid false positives.

## Spec reflection

- **How the spec helped:** the requirement to "communicate uncertainty rather than force
  a binary" drove the entire threshold design. Instead of a single AI/human cutoff, it
  pushed me to define an explicit `uncertain` band and an asymmetric AI threshold, which
  turned out to be the most important safety property of the system.
- **How the implementation diverged:** the spec frames confidence as a single score, and
  an early version reported only `final_p_ai`. I diverged by exposing a separate
  `confidence = max(p_ai, 1−p_ai)` alongside `p_ai`, because "AI probability 0.18" is
  confusing to a reader who is being told the work is *human* — they should see "82%
  confident it's human." The internal `p_ai` still drives all logic; `confidence` is the
  human-facing framing.

## AI usage

- **Directed:** "Generate the second detection signal (stylometric) and the confidence
  scoring that combines both signals per my planning.md thresholds."
  **Produced:** a working repetition signal plus a fusion function.
  **Revised/overrode:** the generated scorer used symmetric thresholds (AI and human at
  equal distance from 0.5), silently diverging from my spec. I corrected it to the
  asymmetric `≥0.80 / <0.40` bands so a borderline `0.55` cannot flag a human as AI.
- **Directed:** "Generate the label-generation function mapping confidence to the three
  label variants, and the `POST /appeal` endpoint."
  **Produced:** label mapping + an appeal handler.
  **Revised/overrode:** the first appeal handler overwrote the original classification
  row when filing an appeal. I rewrote `register_appeal` to mutate **only** `status` and
  `appeal_reasoning`, preserving the original decision so the audit trail stays intact
  for a human reviewer.

## Stretch features (all four implemented)

### 1. Ensemble detection (≥3 signals, documented weighting)
Three signals fused `0.25·burstiness + 0.25·repetition + 0.50·llm`, with the AI verdict
gated on corroboration (the LLM alone maxes at 0.50, below the 0.80 threshold). Detailed
above.

### 2. Analytics dashboard — `GET /analytics`
Aggregates the audit log into detection patterns + platform health. The chosen
additional metric is **appeal rate**, because a rising appeal rate is the earliest
signal the detector is misfiring.

```json
{ "total_submissions": 3,
  "attribution_distribution": {"likely_ai": 1, "uncertain": 0, "likely_human": 2},
  "appeal_count": 0, "appeal_rate": 0.0,
  "avg_confidence": 0.8703, "uncertain_rate": 0.0,
  "modality_distribution": {"text": 1, "image_metadata": 2},
  "verified_creators": 1,
  "avg_signal_scores": {"burstiness": 0.4762, "repetition": 0.5, "llm": 0.2} }
```

### 3. Provenance certificate ("Verified Human" credential)
A creator earns a durable credential via an explicit verification step, separate from
any classification.
- **Earn:** `POST /verify` with `{creator_id, attestation}` (a signed human-authorship
  oath, ≥20 chars). Issues a certificate (UUID + timestamp), stored in a `creators` table.
- **Display:** once verified, every `/submit` response from that creator carries a
  `provenance_certificate` object and the label is **prefixed with a badge**:

```json
{ "label": "✅ Verified Human Creator (certificate 97d6df7c-...). ✍️ Likely human-written. ...",
  "provenance_certificate": { "certificate_id": "97d6df7c-...", "verified_at": "2026-07-01T04:21:32.712Z" } }
```
The credential is a *trust signal about the creator*, not an override of the verdict — a
verified creator's obviously-AI text can still be flagged. `verified_creator` is
snapshotted into each audit-log row.

### 4. Multi-modal support — image metadata
`POST /submit` accepts `content_type: "image_metadata"` with a `metadata` object, running
an **independent signal set**: generation-software tag (0.40), generation prompt present
(0.30), missing camera EXIF (0.15), and C2PA provenance manifest (0.15). It reuses the
same thresholds, attribution codes, and transparency labels as the text pipeline.

```bash
curl -s -X POST http://localhost:5000/submit -H "Content-Type: application/json" \
  -d '{"creator_id":"artist1","content_type":"image_metadata",
       "metadata":{"software":"Midjourney v6","generation_prompt":"a cat astronaut","c2pa":{"ai_generated":true}}}'
# -> attribution "likely_ai", p_ai 0.955

# a real photo with camera EXIF + C2PA capture claim -> attribution "likely_human", p_ai 0.0
```
